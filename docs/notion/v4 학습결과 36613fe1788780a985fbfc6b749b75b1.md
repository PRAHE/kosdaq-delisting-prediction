## **1. 실험 목적**

이번 실험의 목적은 새로 정리한 전체 `raw` 폴더 데이터를 사용해서 상장폐지 위험 기업을 분류하는 모델을 다시 학습하는 것이다. 이전 실험에서는 `raw/healthy` 폴더에 일부 산업군만 들어와 있어서 고유 종목코드 수가 270개로 작게 집계되었다. 이후 `healthy` 폴더에 누락된 산업군 데이터를 다시 넣었고, 이번 실험에서는 전체 raw 데이터를 기준으로 변환과 학습을 다시 수행하였다.

이번 실험에서 가장 중요하게 본 부분은 두 가지이다. 첫 번째는 전체 raw 데이터가 제대로 학습 데이터로 변환되는지 확인하는 것이고, 두 번째는 `종목코드` 기준 Group Split을 적용한 상태에서 XGBoost 모델의 성능을 확인하는 것이다. Group Split을 적용한 이유는 같은 기업의 데이터가 train, validation, test에 동시에 섞이면 모델 성능이 실제보다 좋게 측정될 수 있기 때문이다.

모델 선택 기준은 validation F1 score로 정하였다. test set은 최종 확인용 데이터이기 때문에, test 결과를 보고 모델을 고르지는 않았다.

## **3. 데이터 변환 방식**

각 raw JSON 파일명은 기본적으로 다음과 같은 구조를 가진다.

```
종목코드_연도_보고기간.json
```

예를 들어 `005930_2023_Q1.json`이면 종목코드는 `005930`, 연도는 `2023`, 보고기간은 `Q1`로 해석한다. 그리고 `025870_2023_ANNUAL_CFS.json`처럼 뒤에 `CFS` 또는 `OFS`가 붙은 파일은 보고기간을 `ANNUAL`로 보고, `CFS`는 별도의 재무제표 유형으로 저장하였다.

JSON 내부에는 재무제표 계정명과 금액이 들어 있다. 기존 파이프라인에서 사용하던 계정 추출 로직을 재사용하여 자산, 부채, 자본, 매출액, 매출원가, 영업이익, 당기순이익, 현금및현금성자산, 차입금, 감가상각비 등을 추출하였다. 이후 이 값들을 이용해 부채비율, 유동비율, 매출액순이익률, 총자본회전율, 총자본순이익률 같은 재무비율을 계산하였다.

성장률 계열 변수는 같은 종목코드와 같은 보고기간의 전년도 값과 비교하였다. 예를 들어 `2023_Q1`의 매출액증가율은 `2022_Q1`과 비교하고, `2023_ANNUAL`은 `2022_ANNUAL`과 비교하였다. 분기 데이터와 연간 데이터를 서로 섞어서 비교하면 기간 기준이 달라지기 때문에, 같은 보고기간끼리 비교하도록 설계하였다.

보고기간은 문자열 그대로 넣지 않고 숫자 flag로 바꾸었다.

| feature | 의미 |
| --- | --- |
| `period_Q1` | 1분기 데이터이면 1, 아니면 0 |
| `period_H1` | 반기 데이터이면 1, 아니면 0 |
| `period_Q3` | 3분기 데이터이면 1, 아니면 0 |
| `period_ANNUAL` | 연간 데이터이면 1, 아니면 0 |

최종 feature 수는 64개이다. 여기에는 재무비율 feature, 결측 여부를 나타내는 `_missing` feature, 보고기간 flag feature가 포함된다.

## **4. Group Split 적용 결과**

이번 실험에서는 행 단위로 무작위 분할하지 않고, `종목코드`를 기준으로 Group Split을 적용하였다.

행 단위 split을 사용하면 같은 기업의 다른 연도나 다른 분기 데이터가 train과 test에 동시에 들어갈 수 있다. 예를 들어 어떤 기업의 2022년 데이터가 train에 있고 2023년 데이터가 test에 있으면, 모델이 완전히 새로운 기업을 평가하는 것이 아니라 이미 비슷한 기업 정보를 본 상태에서 평가될 수 있다. 이런 경우 성능이 실제보다 높게 나올 수 있다.

따라서 이번에는 같은 종목코드가 train, validation, test 중 하나에만 들어가도록 나누었다.

| split | 행 수 | label=1 | label=0 | 고유 종목코드 수 |
| --- | --- | --- | --- | --- |
| train | 39,424 | 1,991 | 37,433 | 1,387 |
| validation | 8,330 | 410 | 7,920 | 298 |
| test | 8,479 | 447 | 8,032 | 298 |

split 간 종목코드 중복은 모두 0개였다.

| 비교 | 중복 종목코드 수 |
| --- | --- |
| train-validation | 0 |
| train-test | 0 |
| validation-test | 0 |

즉, 이번 실험은 동일 기업 데이터가 서로 다른 split에 섞이지 않는 조건에서 수행되었다.

## **5. 평가 지표 설명**

이번 실험에서는 여러 지표를 함께 확인하였다.

| 지표 | 의미 |
| --- | --- |
| precision | 모델이 상폐기업이라고 예측한 것 중 실제 상폐기업인 비율 |
| recall | 실제 상폐기업 중 모델이 상폐기업이라고 찾아낸 비율 |
| F1 score | precision과 recall의 균형을 보는 지표 |
| ROC-AUC | 전체 threshold 범위에서 label 0과 label 1을 구분하는 능력 |
| PR-AUC | precision과 recall의 관계를 보는 지표 |

이번 데이터는 전체 56,233행 중 label=1이 2,848행으로, 정상기업(label=0)이 훨씬 많다. 이런 경우 accuracy만 보면 모델이 정상기업 위주로 예측해도 좋아 보일 수 있다. 그래서 이번 실험에서는 F1 score와 PR-AUC를 중요하게 보았다.

## **6. XGBoost 하이퍼파라미터 조합 실험**

하이퍼파라미터 조합 실험은 수행하였다.

이번 실험에서는 XGBoost 설정을 여러 가지로 바꿔보며 validation 성능을 비교하였다. XGBoost는 여러 개의 의사결정나무를 순차적으로 학습하는 모델이다. 이때 모델의 복잡도나 학습 속도 등을 조절하는 값이 하이퍼파라미터이다.

이번에 비교한 주요 하이퍼파라미터는 다음과 같다.

| 파라미터 | 설명 |
| --- | --- |
| `max_depth` | 나무 하나가 얼마나 깊게 분기할 수 있는지 정하는 값 |
| `learning_rate` | 한 번 학습할 때 반영하는 정도를 조절하는 값 |
| `n_estimators` | 사용할 의사결정나무 개수 |
| `scale_pos_weight` | label=1 클래스에 주는 가중치 |
| `subsample` | 각 나무를 학습할 때 사용할 행 샘플 비율 |
| `colsample_bytree` | 각 나무를 학습할 때 사용할 feature 샘플 비율 |

실험 조합은 총 24개였다.

| 구분 | 개수 | 설명 |
| --- | --- | --- |
| baseline | 1 | 기본 XGBoost 설정 |
| 1차 grid 탐색 | 12 | `max_depth`, `learning_rate`, `n_estimators` 조합 비교 |
| scale_pos_weight 조정 | 9 | 1차 상위 3개 조합에 대해 0.75배, 1.0배, 1.25배 비교 |
| feature 제거 실험 | 2 | high-missing 제거, low-importance 제거 |

따라서 하이퍼파라미터 조합 실험은 실제로 수행되었다. 그중 validation F1 기준으로 가장 좋은 설정은 `grid1_02`였다.

`grid1_02`의 설정은 다음과 같다.

| 항목 | 값 |
| --- | --- |
| max_depth | 3 |
| learning_rate | 0.03 |
| n_estimators | 400 |
| subsample | 0.8 |
| colsample_bytree | 0.8 |
| scale_pos_weight | 18.8011 |

`scale_pos_weight`가 큰 이유는 전체 raw 데이터에서 label=1인 상폐기업보다 label=0인 정상기업이 훨씬 많기 때문이다. 이 값은 모델이 소수 클래스인 상폐기업을 너무 무시하지 않도록 가중치를 주는 역할을 한다.

## **7. Threshold 최적화 수행 여부**

threshold 최적화도 수행하였다.

XGBoost는 각 데이터가 label=1일 확률을 출력한다. 일반적으로는 확률이 0.5 이상이면 label=1로 예측하지만, 상폐기업 탐지처럼 precision과 recall의 균형이 중요한 문제에서는 0.5가 최적 기준이 아닐 수 있다.

그래서 이번 실험에서는 각 모델마다 validation set에서 threshold를 `0.05`부터 `0.95`까지 바꿔가며 F1 score를 계산하였다. 그리고 validation F1이 가장 높게 나오는 threshold를 선택하였다. 선택된 threshold는 test set에도 그대로 적용하였다.

주요 실험의 threshold는 다음과 같다.

| 실험 | threshold | validation F1 | test F1 |
| --- | --- | --- | --- |
| baseline_default | 0.6950 | 0.3560 | 0.3212 |
| grid1_02 | 0.7400 | 0.3983 | 0.3315 |
| feature_drop_high_missing | 0.7400 | 0.3983 | 0.3315 |
| feature_drop_low_importance | 0.7600 | 0.3875 | 0.3360 |

즉, threshold 최적화는 아직 안 한 것이 아니라 모든 실험 후보에 대해 수행된 상태이다. 최종 모델인 `grid1_02`는 threshold 0.7400을 사용하였다.

## **8. Feature subset 및 제거 실험**

feature subset 및 제거 실험도 수행하였다.

이번에는 두 가지 방식으로 feature 제거를 시도하였다.

첫 번째는 `drop_high_missing`이다. 결측률이 60% 이상인 feature를 제거하려고 했다. 하지만 실제로 이 기준에 해당하는 feature가 없어서 제거된 feature 수는 0개였다. 그래서 `drop_high_missing` 결과는 `grid1_02`와 동일하게 나왔다.

두 번째는 `drop_low_importance`이다. XGBoost feature importance를 기준으로 중요도가 낮은 feature를 제거하였다. 이 실험에서는 64개 feature 중 27개를 제거했고, 37개 feature만 사용하였다.

feature 제거 실험 결과는 다음과 같다.

| 실험 | feature 수 | 제거 feature 수 | validation F1 | test F1 | test PR-AUC |
| --- | --- | --- | --- | --- | --- |
| grid1_02 | 64 | 0 | 0.3983 | 0.3315 | 0.2895 |
| drop_high_missing | 64 | 0 | 0.3983 | 0.3315 | 0.2895 |
| drop_low_importance | 37 | 27 | 0.3875 | 0.3360 | 0.2890 |

`drop_low_importance`는 test F1만 보면 0.3360으로 `grid1_02`보다 조금 높았지만, validation F1은 0.3875로 낮았다. 이번 실험의 모델 선택 기준은 validation F1이므로, 최종 모델에는 feature 제거를 적용하지 않았다.

최종 모델 선택은 validation F1 기준으로 했지만, test set에서 어떤 모델이 가장 높은 F1을 보였는지도 참고용으로 확인하였다.

전체 raw 기준에서는 이전 270개 종목 기준보다 F1 score가 크게 낮아졌다. 가장 큰 이유는 데이터가 훨씬 커지면서 label 불균형이 강해졌기 때문이다. 전체 학습 행 56,233개 중 label=1은 2,848개로 약 5% 수준이다. 즉, 상폐기업이 정상기업보다 훨씬 적기 때문에 모델이 상폐기업을 안정적으로 잡아내는 것이 더 어려워졌다.

## **12. 이번 실험에서 확인한 점**

이번 실험에서는 하이퍼파라미터 조합 실험, threshold 최적화, feature subset 및 제거 실험을 모두 수행하였다. 정리하면 다음과 같다.

- 하이퍼파라미터 조합 실험은 총 24개 설정을 비교하였다.
- threshold 최적화는 각 모델마다 validation F1 기준으로 수행하였다.
- high-missing feature 제거는 제거 대상이 없어 결과가 동일했다.
- low-importance feature 제거는 27개 feature를 제거했지만 validation F1이 낮아졌다.
- 최종 모델은 validation F1 기준 `grid1_02`로 선택하였다.

이번 결과는 전체 raw 데이터를 넣었을 때 모델 성능이 더 현실적으로 낮아진 사례로 볼 수 있다. 특히 정상기업 데이터가 크게 늘어나면서 label=1 비율이 낮아졌고, 이 때문에 F1과 PR-AUC가 이전보다 낮아졌다.

## **13. 결론**

전체 raw 데이터를 다시 넣은 뒤 학습을 재수행한 결과, 고유 종목코드는 1,983개로 정상적으로 증가하였다. Group Split 검증에서도 train, validation, test 사이 종목코드 중복은 0개로 확인되었다.

XGBoost 하이퍼파라미터 탐색 결과 validation F1 기준 best setting은 `grid1_02`였고, validation F1은 0.3983, test F1은 0.3315, test PR-AUC는 0.2895였다. 전체 데이터 기준에서는 label 불균형이 강해져 성능이 낮아졌지만, 더 넓은 데이터 범위에서 평가한 결과이므로 이전보다 더 현실적인 성능으로 해석할 수 있다.

다음 실험에서는 상폐기업 비율이 낮아진 문제를 더 직접적으로 다루는 방향이 필요하다. 예를 들어 threshold를 F1뿐만 아니라 recall 중심으로도 비교하거나, `scale_pos_weight` 탐색 범위를 더 넓히거나, 산업군별 분포 차이를 고려한 split 또는 feature 검토가 필요하다.

## **14. 다음 실험 방향**

다음 실험의 목표는 단순히 실험 개수를 많이 늘리는 것이 아니라, 현재 결과에서 F1 score가 낮게 나온 원인을 조금 더 직접적으로 다루는 것이다. 이번 전체 raw 기준 데이터에서는 label=1인 상폐기업 비율이 약 5% 수준으로 낮았다. 따라서 모델이 정상기업을 많이 맞히는 것만으로는 좋은 모델이라고 보기 어렵고, 상폐기업을 얼마나 잘 찾아내는지가 중요하다. F1 score는 precision과 recall을 함께 보는 지표이기 때문에, 다음 실험에서는 이 두 값의 균형을 조정하는 방향이 필요해보인다.

첫 번째로 threshold 선택 방식을 더 다양하게 비교해볼 필요가 있다. 현재 실험에서는 validation set에서 F1 score가 가장 높아지는 threshold를 선택하였다. 이 방식은 기본적으로 적절하지만, 상폐기업 탐지에서는 recall이 너무 낮아지면 실제 위험 기업을 놓칠 수 있다. 따라서 다음에는 단순히 F1이 가장 높은 threshold만 고르지 않고, recall을 일정 수준 이상으로 유지한 상태에서 precision을 높이는 방식도 함께 비교해볼 수 있다. 예를 들어 recall이 0.45 이상 또는 0.50 이상인 threshold 중에서 precision이 가장 높은 값을 선택하는 식으로 실험할 수 있다.

두 번째로 `scale_pos_weight` 탐색 범위를 조금 더 넓혀볼 필요가 있다. `scale_pos_weight`는 label=1처럼 개수가 적은 클래스에 더 큰 가중치를 주는 XGBoost의 하이퍼파라미터이다. 현재 데이터에서는 정상기업이 상폐기업보다 훨씬 많기 때문에, 모델이 상폐기업을 덜 적극적으로 예측할 가능성이 있다. 따라서 현재 기준보다 조금 더 큰 가중치를 주는 실험을 해보면 recall이 올라갈 수 있다. 다만 가중치를 너무 크게 주면 상폐기업으로 잘못 예측하는 경우가 늘어 precision이 떨어질 수 있으므로, `0.5x`, `1.0x`, `1.5x`, `2.0x` 정도의 범위에서 비교하는 것이 현실적이라고 판단된다.

세 번째로 feature 제거 실험은 한 번에 많이 제거하기보다 단계적으로 진행하는 것이 좋을 것 같다. 이번 low-importance feature 제거 실험에서는 27개 feature를 한 번에 제거하였다. test F1은 약간 좋아졌지만 validation F1은 낮아졌기 때문에 최종 모델로 선택하지 않았다. 다음에는 중요도가 낮은 feature를 하위 5개, 10개, 15개처럼 단계적으로 제거하면서 성능 변화를 확인하는 방식이 더 안정적일 수 있다. 이렇게 하면 어느 정도까지 feature를 줄여도 성능이 유지되는지 확인할 수 있고, 너무 많이 제거했을 때 성능이 떨어지는 지점도 볼 수 있다.

정리하면 다음 실험은 새로운 데이터를 더 모으기보다는, 현재 전체 raw 데이터 안에서 threshold, class weight, feature subset, 오류 분석을 중심으로 진행하는 것이 현실적이다. 특히 2주 안에 혼자 수행하는 실험이라는 점을 고려하면, 우선순위는 threshold 선택 방식 비교, `scale_pos_weight` 범위 확장, 단계적 feature 제거 순서로 두는 것이 적절하다고 생각한다.
