# **상폐일 기준 540일 이내 라벨 재정의 XGBoost 실험 결과**

## **1. 실험 목적**

이번 실험은 상장폐지 예측 문제에 더 맞는 라벨 기준을 찾기 위해 진행하였다. 이전에는 상폐일 전 365일 이내를 `label=1`로 두는 1차 실험과, 상폐일 전 730일 이내를 `label=1`로 두는 2차 실험을 수행하였다. 365일 기준은 상폐 직전 예측이라는 의미는 강하지만 `label=1` 데이터가 너무 적었고, 730일 기준은 성능은 더 좋아졌지만 상폐 직전이라기보다는 상폐 전 약 2년 이내 위험 예측에 가까웠다.

그래서 이번 실험에서는 두 기준의 중간값으로 **상폐일 전 540일 이내를 `label=1`**로 정의하였다. 540일은 약 1년 6개월에 해당하므로, 상폐 직전성은 어느 정도 유지하면서도 365일 기준보다 더 많은 positive 데이터를 확보할 수 있는지 확인하는 목적이다.

## **2. 라벨 정의 방식**

이번 실험의 라벨 기준은 다음과 같다.

| 조건 | label |
| --- | --- |
| healthy 기업 전체 | 0 |
| delisted 기업이고 보고기간 기준일이 상폐일 전 1~540일 이내 | 1 |
| delisted 기업이고 보고기간 기준일이 상폐일 전 541일 이상 | 0 |
| 보고기간 기준일이 상폐일 이후 | 제외 |

여기서 보고기간 기준일은 실제 공시일이 아니라 재무제표의 보고기간 종료일로 두었다. raw JSON에는 실제 공시일이 따로 없기 때문에, 예측 시점의 기준을 보고기간 종료일로 통일하였다.

| 보고기간 | 기준일 |
| --- | --- |
| Q1 | 해당 연도 3월 31일 |
| H1 | 해당 연도 6월 30일 |
| Q3 | 해당 연도 9월 30일 |
| ANNUAL | 해당 연도 12월 31일 |

예를 들어 어떤 기업의 상폐일이 2024년 12월 31일이라면, 2023년 7월 이후의 보고기간은 대체로 540일 이내에 들어올 수 있다. 반대로 상폐일까지 540일보다 더 많이 남은 과거 데이터는 아직 상폐 위험이 직접적으로 가까운 시점은 아니라고 보고 `label=0`으로 두었다.

## **3. 사용 데이터와 상폐일 매칭**

이번 실험은 보강된 `First_Data/2015_2025_delisted.csv`를 사용하였다. 이 파일에는 기존에 상폐일이 없었던 78개 기업의 상폐일이 추가되어 있었고, `raw/delisted`에 있는 170개 종목코드가 모두 CSV와 매칭되었다.

| 항목 | 값 |
| --- | --- |
| raw/delisted 고유 종목코드 수 | 170 |
| 상폐일 CSV와 매칭된 종목 수 | 170 |
| 미매칭 종목 수 | 0 |
| 상폐일 형식 오류 | 0 |

다만 170개 종목이 모두 최종 학습 행에 남은 것은 아니다. 일부 행은 보고기간 기준일이 상폐일 이후였기 때문에 제외하였다. 예측 모델에서는 이미 상폐된 이후의 데이터를 사용할 수 없기 때문이다.

기존 전체 raw feature CSV인 `raw_모델학습용_20260521_fullraw.csv`를 재사용하고, 라벨만 540일 기준으로 다시 부여하였다. 따라서 재무비율 계산 방식은 이전 fullraw 실험과 동일하며, 이번 실험에서 달라진 핵심은 label 정의이다.

## **6. 실험 구성**

이번 실험에서는 XGBoost 모델을 사용하였다. 기본 baseline만 본 것이 아니라, 하이퍼파라미터와 threshold 전략을 함께 비교하였다.

| 실험 구분 | 내용 |
| --- | --- |
| baseline | 기본 XGBoost 설정 |
| grid 탐색 | `max_depth`, `learning_rate`, `n_estimators` 조합 비교 |
| scale_pos_weight 확장 | 0.5x, 1.0x, 1.5x, 2.0x, 2.5x, 3.0x 비교 |
| threshold 전략 | `f1_max`, `recall_floor_0.30`, `0.35`, `0.40`, `0.45`, `0.50` 비교 |
| feature subset | all_features, low-importance 5개 제거, low-importance 10개 제거 비교 |

최종 모델은 validation F1 기준으로 선택하였다. test set은 최종 확인용으로만 사용했고, test 결과를 보고 모델을 다시 선택하지 않았다.

## **8. 최종 선택 모델 성능**

최종 선택 모델의 성능은 다음과 같다.

| dataset | precision | recall | F1 | ROC-AUC | PR-AUC |
| --- | --- | --- | --- | --- | --- |
| validation | 0.1486 | 0.2857 | 0.1956 | 0.8162 | 0.0731 |
| test | 0.1629 | 0.3412 | 0.2205 | 0.9283 | 0.1794 |

test F1은 0.2205로 나왔다. 365일 기준 test F1 0.0426보다는 많이 개선되었지만, 730일 기준 test F1 0.2874보다는 낮았다. 즉, 540일 기준은 365일 기준보다 positive 샘플을 늘리는 효과는 있었지만, 이번 데이터에서는 730일 기준만큼 안정적인 성능을 만들지는 못했다.

## **11. 365일, 540일, 730일 비교**

| 기준 | label=1 행 수 | validation F1 | test recall | test F1 | test PR-AUC |
| --- | --- | --- | --- | --- | --- |
| 365일 | 275 | 0.2545 | 0.0233 | 0.0426 | 0.0978 |
| 540일 | 545 | 0.1956 | 0.3412 | 0.2205 | 0.1794 |
| 730일 | 740 | 0.2092 | 0.4138 | 0.2874 | 0.2261 |

540일 기준은 365일 기준보다 test recall과 test F1이 크게 좋아졌다. 하지만 730일 기준과 비교하면 label=1 수가 더 적고, test F1과 PR-AUC도 낮았다. 따라서 현재까지는 730일 기준이 가장 안정적인 label window로 보인다.

## **12. 결론**

이번 실험은 **상폐일 전 540일 이내를 `label=1`로 두고 학습한 실험**이다. 540일 기준은 365일 기준보다 positive 데이터를 더 많이 확보할 수 있었고, 그 결과 test F1도 0.0426에서 0.2205로 개선되었다. 하지만 730일 기준의 test F1 0.2874에는 미치지 못했다.

따라서 540일 기준은 중간 실험으로 의미가 있지만, 현재 데이터에서는 `상폐일 전 730일 이내 = label 1` 기준이 더 안정적인 결과를 보였다. 다음 단계에서는 730일 기준을 유지한 상태에서 threshold와 `scale_pos_weight`를 더 세밀하게 조정하거나, validation F1뿐 아니라 validation PR-AUC를 함께 고려하는 선택 기준을 검토하는 것이 좋다.
