# **상폐일 기준 365일 이내 라벨 재정의 XGBoost 1차 실험 결과**

## **1. 실험 목적**

이번 실험은 기존의 `delisted=1`, `healthy=0` 라벨 방식이 실제 상폐 예측 문제와 완전히 맞지 않을 수 있다는 문제의식에서 시작하였다. 기존 방식은 상폐기업의 모든 연도와 모든 보고기간을 `label=1`로 두었기 때문에, 엄밀히 말하면 "이 기업이 상폐기업 집단에 속하는가"를 구분하는 성격이 강했다.

하지만 실제로 하고 싶은 것은 특정 시점의 재무제표를 보고 "앞으로 상폐될 위험이 있는가"를 예측하는 것이다. 그래서 이번 1차 실험에서는 상폐일이 확인되는 기업만 사용하여, 상폐일 전 365일 이내의 보고기간만 `label=1`로 다시 정의하였다. 즉, 상폐 직전의 위험 신호를 모델이 학습하도록 문제를 더 엄격하게 바꾸었다.

## **2. 라벨 재정의 방식**

이번 실험에서는 `First_Data/2015_2025_delisted.csv`에 상폐일이 존재하는 delisted 종목만 사용하였다.

| 항목 | 값 |
| --- | --- |
| raw/delisted 고유 종목코드 수 | 170 |
| 상폐일 CSV와 매칭된 종목 수 | 92 |
| 상폐일이 없어 제외한 종목 수 | 78 |
| 최종 사용된 delisted 종목 수 | 85 |

매칭된 92개 중에서도 일부 행은 보고기간 종료일이 상폐일 이후였기 때문에 제외하였다. 예측 모델에서는 상폐 이후의 재무제표를 사용할 수 없기 때문이다.

보고기간 기준일은 다음처럼 정하였다.

| 보고기간 | 기준일 |
| --- | --- |
| Q1 | 해당 연도 3월 31일 |
| H1 | 해당 연도 6월 30일 |
| Q3 | 해당 연도 9월 30일 |
| ANNUAL | 해당 연도 12월 31일 |

라벨 규칙은 다음과 같다.

| 조건 | label |
| --- | --- |
| healthy 기업 전체 | 0 |
| delisted 기업이지만 상폐일까지 365일 초과 | 0 |
| delisted 기업이고 상폐일까지 1~365일 | 1 |
| 보고기간 기준일이 상폐일 이후 | 제외 |
| 상폐일 CSV에 없는 delisted 종목 | 제외 |

기존 전체 raw feature CSV를 기준으로 라벨만 재정의하였다. 재무비율 feature는 전체 raw 데이터에서 이미 계산된 값을 사용했고, 이번 실험의 핵심은 라벨 정의를 상폐일 기준으로 바꾸는 것이었다.

## **5. 수행한 실험**

이번 1차 실험에서는 총 29개 모델 설정을 학습했고, 각 설정마다 3가지 threshold 전략을 비교하였다. validation/test를 각각 저장했기 때문에 metrics CSV에는 총 174개 행이 기록되었다.

| 실험 구분 | 개수 | 설명 |
| --- | --- | --- |
| baseline | 1 | 기본 XGBoost 설정 |
| 1차 grid | 12 | `max_depth`, `learning_rate`, `n_estimators` 조합 |
| scale_pos_weight 확장 | 12 | 상위 3개 조합에 대해 `0.5x`, `1.0x`, `1.5x`, `2.0x` 비교 |
| feature 제거 | 4 | high-missing 제거, 중요도 하위 5/10/15개 제거 |

threshold 전략은 다음 세 가지를 비교하였다.

| threshold 전략 | 의미 |
| --- | --- |
| f1_max | validation F1이 가장 높은 threshold 선택 |
| recall_floor_0.45 | validation recall 0.45 이상을 만족하는 값 중 precision 우선 |
| recall_floor_0.50 | validation recall 0.50 이상을 만족하는 값 중 precision 우선 |

## **6. Validation 기준 상위 결과**

최종 모델 선택은 validation F1 기준으로 진행하였다.

| 순위 | 실험명 | feature set | threshold 전략 | threshold | precision | recall | F1 | ROC-AUC | PR-AUC |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | grid1_01 | all_features | f1_max | 0.950 | 0.4667 | 0.1750 | 0.2545 | 0.9387 | 0.1789 |
| 2 | scale_grid1_01_1.00x | all_features | f1_max | 0.950 | 0.4667 | 0.1750 | 0.2545 | 0.9387 | 0.1789 |
| 3 | feature_drop_high_missing | drop_high_missing | f1_max | 0.950 | 0.4667 | 0.1750 | 0.2545 | 0.9387 | 0.1789 |
| 4 | scale_grid1_02_0.50x | all_features | f1_max | 0.930 | 0.4667 | 0.1750 | 0.2545 | 0.9400 | 0.1769 |
| 5 | scale_grid1_02_1.50x | all_features | f1_max | 0.945 | 0.3077 | 0.2000 | 0.2424 | 0.9370 | 0.1874 |

validation 기준 best setting은 `grid1_01`이었다. threshold가 0.950으로 높게 선택된 것은, validation set에서 양성 예측을 매우 조심스럽게 했을 때 precision이 높아져 F1이 가장 높게 나온 결과이다.

## **7. Test set 결과**

validation F1 기준으로 선택된 `grid1_01`을 test set에 그대로 적용한 결과는 다음과 같다.

| dataset | precision | recall | F1 | ROC-AUC | PR-AUC |
| --- | --- | --- | --- | --- | --- |
| validation | 0.4667 | 0.1750 | 0.2545 | 0.9387 | 0.1789 |
| test | 0.2500 | 0.0233 | 0.0426 | 0.9357 | 0.0978 |

test F1이 매우 낮게 나온 가장 큰 이유는 recall이 0.0233으로 낮았기 때문이다. 즉, test set의 실제 positive 43개 중 모델이 잡아낸 양성은 매우 적었다. 반면 ROC-AUC는 0.9357로 높게 나왔기 때문에, 확률 순위 자체는 어느 정도 구분력을 가지고 있지만 최종 threshold가 너무 보수적으로 잡히면서 실제 양성 예측 수가 적어진 것으로 해석할 수 있다.

## **10. 결론**

이번 1차 실험은 F1 score 자체를 높이는 것보다, 상폐 예측 문제에 더 적합한 라벨 정의를 적용했다는 점이 중요하다. 기존에는 상폐기업 전체 기간을 `label=1`로 두었기 때문에 모델이 상폐기업과 정상기업의 집단 차이를 학습했을 가능성이 있었다. 이번에는 상폐일 전 365일 이내 데이터만 `label=1`로 두어, 특정 시점에서 향후 상폐 위험을 예측하는 구조에 더 가까워졌다.

결과적으로 validation F1은 0.2545, test F1은 0.0426으로 낮게 나왔다. 특히 test recall이 낮아 실제 상폐 직전 데이터를 많이 놓쳤다. 하지만 ROC-AUC가 높게 나온 점을 보면, 모델의 확률 순위에는 어느 정도 정보가 있고 threshold 선택이 너무 보수적이었을 가능성이 있다.

다음 2차 실험에서는 사용자가 직접 보강한 78개 종목의 상폐일을 추가한 뒤, positive label 수를 늘려 같은 실험을 다시 수행하는 것이 좋다. 또한 이번 test 결과에서 `scale_pos_weight` 2.0배 설정이 더 높은 F1을 보였으므로, 다음에는 class weight 탐색 범위를 더 넓히고 recall 중심 threshold 전략을 더 적극적으로 비교할 필요가 있다.
